“打鼓声”（类似直升机螺旋桨的“突突突”声）在音频流开发中非常典型。这通常不是音频数据本身错了，而是**数据流不连续**或**错误的填充（Padding）**造成的。

简单来说：你的播放器在播放一段音频，然后播放了一小段静音（或空数据），然后又播放音频。这种“有声-无声-有声-无声”的循环，听起来就是打鼓声。

### 核心原因诊断

1. **强制填充 0 (Zero Padding)**：

- **现象**：WebSocket 每次发过来的包大小是不固定的（比如有时候是 200 字节，有时候是 1000 字节）。
- **错误做法**：如果你的代码逻辑是“必须凑够 4096 字节才发送”，而当数据不足时，你往后面**补了 0**。
- **后果**：每收到一个网络小包，你就播放一段声音+一段静音。这就是“打鼓”。

2. **采样点错位 (Byte Alignment)**：

- **现象**：PCM 16bit 音频是 **2 个字节** 代表一个声音点。
- **错误做法**：如果在处理数据流时，不小心把数据切断在奇数位置（比如第 101 个字节），后续所有的字节都会错位（高低位互换），产生巨大的爆破音或周期性噪音。

---

### 解决方案：使用“残余缓冲区” (Residue Buffer)

你需要引入一个**累积缓冲区**。收到数据先扔进缓冲区，**只有当缓冲区填满一个完整的 chunk 时，才切出来发送**，剩下的留给下一次。绝对不要补 0（除非是整个会话彻底结束）。

请用下面的代码替换类中的 `_push_audio_chunk` 和 `__init__` 部分：

#### 1. 修改 `__init__` 初始化缓冲区

在 `__init__` 方法中增加一个变量来存储上次没发完的数据：

```python
def __init__(self, opt, parent):
    super().__init__(opt, parent)
    # ... 其他原有代码 ...

    # 【新增】用于存储流式数据中不足一个chunk的剩余部分
    self.audio_buffer = np.array([], dtype=np.float32)

```

#### 2. 重写 `_push_audio_chunk` (核心修复)

这个版本解决了打鼓声问题：它会把碎片数据拼起来，凑够一帧再发，**绝不填充静音**。

```python
def _push_audio_chunk(self, audio_array, eventpoint):
    """
    修复版：使用累积缓冲区，消除打鼓声
    """
    # 1. 将新收到的数据拼接到暂存区
    if len(self.audio_buffer) > 0:
        self.audio_buffer = np.concatenate((self.audio_buffer, audio_array))
    else:
        self.audio_buffer = audio_array

    # 2. 循环切片：只要暂存区的数据够一个 chunk，就切出来发送
    while len(self.audio_buffer) >= self.chunk:
        # 切出一个完整的 chunk
        chunk_data = self.audio_buffer[:self.chunk]
        # 更新暂存区，保留剩余部分
        self.audio_buffer = self.audio_buffer[self.chunk:]

        # 发送这个完整的 chunk
        if getattr(self, 'direct_to_webrtc', False):
            self._send_to_webrtc(chunk_data, eventpoint)
        else:
            self.parent.put_audio_frame(chunk_data, eventpoint)

        # 为了避免事件重复发送，后续的切片不再携带 eventpoint
        eventpoint = {}

    # 3. 如果剩下了数据（不足一个chunk），就留在 self.audio_buffer 里等待下一次网络包
    # 绝对不要在这里补 0 发送！

```

#### 3. 修改 WebSocket 结束处的逻辑

只有在 `ws.close()` 之后，音频彻底结束了，才处理最后剩下的那一点点尾巴（这时候才允许补 0 或者直接发不完整的帧）。

### 其他可能导致“打鼓声”的检查项

如果上述代码修改后还有问题，请检查以下两点：

1. **采样率不匹配 (Sample Rate Mismatch)**

- 火山引擎请求的是 `16000` (16k)。
- **检查**: 你的 WebRTC 或播放器（Player）初始化时，设置的采样率也是 `16000` 吗？
- 如果你把 16k 的数据喂给 44.1k 的播放器，声音会变得极其快且尖锐；反之则极其慢且低沉。如果采样率微小不匹配（比如 16000 vs 16000，但时钟源不同），会产生周期性的咔哒声，但通常不是剧烈的打鼓声。

2. **WebRTC 的 `AudioFrame` 时间戳**

- 如果你是推流到 RTC，WebRTC 极其依赖**时间戳 (pts)**。
- 在 `_send_to_webrtc` 中，你使用了 `AudioFrame.from_ndarray`。
- **修改建议**: 显式设置 `sample_rate` 和 `pts` (如果有必要)。通常 PyAV 会自动处理，但确保你发送的每一帧 `chunk` 大小是固定的（上面的代码已经保证了这点），这对 WebRTC 的 Jitter Buffer 非常重要。

**总结**：加上那个 `self.audio_buffer` 的逻辑，把碎片拼成整块再发，99% 的概率能解决你的打鼓声。
